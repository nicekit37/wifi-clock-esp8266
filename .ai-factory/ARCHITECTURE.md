# Architecture: Layered Monolith (Firmware Modules)
+
+## Overview
+Этот проект — прошивка для ESP8266 с одной точкой входа (`setup()`/`loop()`), небольшим объёмом кода и ограниченными ресурсами микроконтроллера. Для таких проектов лучше всего подходит упрощённая **слойная архитектура** внутри монолита: разделение кода на уровни отображения (UI), сервисной логики (time/weather/alert), инфраструктуры (WiFi, HTTP, файловая система) и конфигурации, при этом всё остаётся в одной прошивке.
+
+## Decision Rationale
+- **Project type:** Встроенные WiFi‑часы на ESP8266 с TFT‑дисплеем, веб‑интерфейсом и интеграциями с внешними API.
+- **Tech stack:** C++ (Arduino, PlatformIO), ESP8266, LittleFS, TFT_eSPI, ESP8266WebServer, ArduinoJson.
+- **Key factor:** Небольшой объём кода и один бинарник прошивки → важнее простота и читаемое разделение зон ответственности, чем сложные архитектурные паттерны.
+
+Выбранная архитектура: **Layered Architecture внутри монолитной прошивки**:
+- UI‑слой (отрисовка экрана, шрифты, иконки).
+- Сервисный слой (логика времени, погоды, статуса тревоги, форматирование строк).
+- Инфраструктурный слой (WiFi, HTTP‑клиент, NTP, файловая система LittleFS).
+- Слой конфигурации (глобальные настройки, чтение/запись в LittleFS, параметры для веб‑интерфейса).
+
+## Folder Structure
+Текущая и рекомендуемая структура (эволюционная, без жёсткого требования немедленного рефакторинга):
+
+```
+.
+├─ src/
+│  ├─ main.cpp              # Точка входа, инициализация, цикл loop()
+│  ├─ Robot20.h / 35.h / 60.h
+│  │                        # Графика/шрифты для UI‑слоя
+│  ├─ ui/                   # (опционально) функции отрисовки экрана
+│  │  ├─ clock_view.h/.cpp  # часы/дата/день недели
+│  │  ├─ weather_view.h/.cpp# температура, влажность, иконка
+│  │  └─ alert_view.h/.cpp  # статус тревоги, индикаторы
+│  ├─ services/             # (опционально) доменная логика
+│  │  ├─ time_service.h/.cpp    # работа со временем, NTP, форматирование дат
+│  │  ├─ weather_service.h/.cpp # HTTP‑запросы к OpenWeatherMap, парсинг JSON
+│  │  └─ alert_service.h/.cpp   # HTTP‑запросы к API тревоги, интерпретация ответа
+│  └─ infra/                # (опционально) инфраструктура и адаптеры
+│     ├─ wifi_manager.h/.cpp    # подключение к WiFi, мониторинг статуса
+│     ├─ http_client.h/.cpp     # обёртка над WiFiClient/WiFiClientSecure
+│     └─ storage.h/.cpp         # работа с LittleFS и файлами конфигурации
+├─ lib/
+│  └─ User_Setup.h          # Аппаратные настройки TFT_eSPI
+└─ .ai-factory/
+   ├─ DESCRIPTION.md        # Описание проекта и стека
+   └─ ARCHITECTURE.md       # Этот документ
+```
+
+Новые папки (`ui/`, `services/`, `infra/`) можно добавлять по мере роста проекта, постепенно вынося из `main.cpp` крупные куски логики.
+
+## Dependency Rules
+Базовые правила зависимостей внутри прошивки:
+
+- ✅ `ui` может зависеть от `services` (UI вызывает сервисы, но не наоборот).
+- ✅ `services` может зависеть от `infra` (сервисы используют WiFi, HTTP, файловую систему).
+- ✅ Любой слой может читать глобальную конфигурацию/константы из `main.h` (по соглашению проекта).
+- ❌ `infra` не должен зависеть от `ui` (инфраструктура не знает про отрисовку).
+- ❌ `infra` не должен зависеть от конкретных экранов или форматов вывода.
+- ❌ Внутри сервисов избегать прямой работы с низкоуровневым UI (например, прямые вызовы `tft.draw...`).
+
+В текущем состоянии часть логики уже смешана в `main.cpp`. Эти правила описывают **целевое направление эволюции**: при добавлении новой функциональности стараться соблюдать границы слоёв и постепенно выносить старый код.
+
+## Layer/Module Communication
+Рекомендуемые паттерны взаимодействия:
+
+- UI‑слой вызывает функции сервисного слоя и получает уже подготовленные данные (строки, структуры с полями `temp`, `humidity`, `icon` и т.п.).
+- Сервисы инкапсулируют детали HTTP‑запросов, разбор JSON и преобразование в удобные структуры `WeatherData`, `ForecastDay`, `AlertData`.
+- Инфраструктура предоставляет небольшие, стабильные обёртки:
+  - `bool wifiEnsureConnected()` — подключение/повторное подключение к WiFi.
+  - `bool storageLoadConfig(Config& cfg)` / `bool storageSaveConfig(const Config& cfg)` — работа с LittleFS.
+  - `bool httpGetJson(const String& url, String& outJson)` — общий способ сделать GET‑запрос.
+
+## Key Principles
+1. **Чёткие зоны ответственности:** UI отвечает только за отображение, сервисы — за бизнес‑логику, инфраструктура — за низкоуровневую работу с железом и сетью.
+2. **Минимизация дублирования:** общие операции (например, HTTP‑GET с таймаутами и обработкой ошибок) выносятся в одну функцию/модуль и переиспользуются.
+3. **Пошаговый рефакторинг:** не требуется одномоментно переписывать `main.cpp`, достаточно упорядочивать код при добавлении/изменении функций.
+4. **Читаемость важнее абстракций:** на микроконтроллере лучше немного более простой, но предсказуемый код, чем чрезмерно усложнённые слои.
+
+## Code Examples
+
+### Пример: сервис погоды поверх инфраструктуры HTTP
+
+```cpp
+// infra/http_client.h
+bool httpGetJson(const String& url, String& outJson);
+
+// services/weather_service.h
+struct WeatherInfo {
+  float temperature;
+  int humidity;
+  String description;
+};
+
+bool fetchCurrentWeather(WeatherInfo& out);

+
+UI‑слой использует только `WeatherInfo` и не знает, как именно выполняется HTTP‑запрос или парсинг JSON.

+## Anti-Patterns
+- ❌ Разрастание `loop()` огромным количеством прямых HTTP‑запросов и парсинга JSON.
+- ❌ Смешивание логики времени/погоды с низкоуровневыми вызовами `tft` в одних и тех же функциях.
+- ❌ Дублирование кода подключения к WiFi и повторных попыток в разных местах.
+- ❌ Жёсткая привязка сервисов к конкретным глобальным переменным, если их можно передать параметрами.
